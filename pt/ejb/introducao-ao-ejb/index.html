Configurando EJB com JBoss AS 7.1 #

1. Visão geral

Neste artigo vamos discutir como começar com Enterprise JavaBeans (EJB). Usaremos o JBoss AS 7.1.1.Final, mas você pode usar qualquer servidor de sua escolha.

2. Dependências Maven para Bean

Para usar o EJB, certifique-se de adicionar a versão mais recente dele à seção de dependências do seu arquivo pom.xml:

<dependency>
    <groupId>org.jboss.spec.javax.ejb</groupId>
    <artifactId>jboss-ejb-api_3.2_spec</artifactId>
    <version>1.0.0.Final</version>
</dependency>

Certifique-se de adicionar as dependências do JBoss corretamente, pois usaremos o JBoss como nosso servidor de aplicativos neste tutorial. Na parte posterior do tutorial, discutiremos em detalhes sobre como configurar a construção do maven para o projeto.

3. EJB Remoto

Vamos primeiro criar a interface do Bean chamada HelloWorldRemote.

public interface HelloWorldRemote {
    public String getHelloWorld();
}

Agora vamos implementar a interface acima e nomeá-la como HelloWorldBean.

@Stateless
public class HelloWorldBean implements HelloWorldRemote {

    public HelloWorldBean() {
    
    }
    
    @Override
    public String getHelloWorld(){
        return ("Hello World");
    }
}

Observe a notação @Stateless no topo da declaração da classe. Denota um bean de sessão sem estado.

4. Configuração do Maven para Bean Remoto

Nesta seção, discutiremos como configurar o maven para construir e executar o aplicativo no servidor.

Vejamos os plugins um por um.

4.1. Plugin do compilador

O maven-compiler-plugin é usado para compilar as fontes do nosso projeto.

Aqui usamos a versão 2.3.1 do plugin com o JDK de origem e destino definido como 1.7 na configuração.

Definimos essas configurações como propriedades dentro da tag e as referenciamos por meio de ${property}.

<version.compiler.plugin>2.3.1</version.compiler.plugin>
<!-- maven-compiler-plugin -->
<maven.compiler.target>1.7</maven.compiler.target>
<maven.compiler.source>1.7</maven.compiler.source>

4.2 Plugin EJB

Este plugin gera o arquivo Bean, bem como o jar do cliente associado.

Especificamos a versão do ejb como 3.2 e a propriedade generateClient é definida como true, o que gera o cliente.

4.3 Implantando no JBoss

O jboss-as-maven-plugin é usado para implantar, reimplantar, remover implantação ou executar o aplicativo no JBoss AS 7.

Nesta configuração, especificamos o nome do arquivo de compilação igual ao nome do arquivo de compilação do projeto que, por padrão, está no formato artefatoid-versão em nosso caso ejb-remote-1.0-SNAPSHOT.

4.4 Dependências Maven Necessárias para EJB

jboss-javaee-6.0 define a versão das APIs Java EE 6 do JBoss que queremos usar.

A JBoss distribui um conjunto completo de APIs Java EE 6, incluindo uma lista de materiais (BOM).

Uma BOM especifica as versões de uma pilha (ou coleção) de artefatos. Especificamos isso na tag para que sempre tenhamos as versões corretas dos artefatos. O tipo dessa dependência é um pom que contém as dependências necessárias.

<dependency>
    <groupId>org.jboss.spec</groupId>
    <artifactId>jboss-javaee-6.0</artifactId>
    <version>${version.org.jboss.spec.jboss.javaee.6.0}</version>
    <type>pom</type>
    <scope>import</scope>
</dependency>

4.5 Anotações

O seguinte obterá a dependência de anotações:

<dependency>
    <groupId>org.jboss.spec.javax.annotation</groupId>
    <artifactId>jboss-annotations-api_1.1_spec</artifactId>
    <scope>provided</scope>
</dependency>

4.6 EJB versão 3.2

No trecho de código a seguir, obtemos a versão mais recente das especificações:

<dependency>
    <groupId>org.jboss.spec.javax.ejb</groupId>
    <artifactId>jboss-ejb-api_3.2_spec</artifactId>
    <version>1.0.0.Final</version>
</dependency>

Para executar o projeto acima em um servidor JBoss, precisamos primeiro executar:

mvn clean install

Em seguida, precisamos implantá-lo em um servidor JBoss em execução executando o seguinte comando maven:

jboss-as:deploy

Agora você deve ver o arquivo jar sendo implantado no servidor jboss.

Alternativamente, você pode copiar o jar disponível da pasta de destino no projeto e colá-lo na pasta webapp do servidor.

5. Configurando o projeto do cliente

Após criar o bean remoto, devemos testar o bean implantado criando um cliente.

Primeiro vamos discutir a configuração do maven para o projeto.

Plugins Maven 5.1 usados

O maven-compiler-plugin é usado para compilar as fontes do seu projeto.

Especificamos a versão jdk 1.7 para classes de origem e destino.

Nosso cliente é um programa Java, para executá-lo usamos o exec-maven-plugin que ajuda a executar programas do sistema e Java. Precisamos especificar o executável (ou seja, java), classpath e a classe java (com.baeldung.ejb.client.Client).

O caminho de classe é deixado vazio porque o plug-in inclui os argumentos de caminho de classe necessários com base nas dependências fornecidas.

5.2 Dependências Maven para cliente EJB3

Para executar o cliente EJB3, precisamos incluir as seguintes dependências.

Dependemos das interfaces de negócios remotas EJB deste aplicativo para executar o cliente. Portanto, precisamos especificar a dependência do jar do cliente ejb. A tag com valor “ejb-client” é usada para especificar a dependência deste projeto no jar do cliente EJB.

<dependency>
    <groupId>com.theopentutorials.ejb3</groupId>
    <artifactId>ejbmavendemo</artifactId>
    <type>ejb-client</type>
    <version>${project.version}</version>
</dependency>

As dependências jboss-transaction-api_1.1_spec, jboss-ejb-api_3.1_spec, jboss-ejb-client, xnio-api, xnio-nio, jboss-remoting, jboss -sasl, jboss-marshalling-river têm escopo como tempo de execução porque são tempo de execução necessário e não durante o tempo de compilação.

As dependências jboss-javaee-6.0 e jboss-as-ejb-client-bom em dependencyManagement têm escopo como importação. Isso é usado para incluir informações de gerenciamento de dependência de um POM remoto no projeto atual. Esses POMs remotos são fornecidos pelo JBoss que contém as dependências necessárias para executar o cliente.

5.3 Propriedades do cliente JBoss EJB

Crie um arquivo em “src/main/resources” e nomeie-o como jboss-ejb-client.properties.

remote.connectionprovider.create.options.org.xnio.Options.SSL_ENABLED=false
remote.connections=default
remote.connection.default.host=localhost
remote.connection.default.port = 4447
remote.connection.default.connect.options.org.xnio.Options.SASL_POLICY_NOANONYMOUS=false

6. Criando a classe de cliente

Primeiro criamos uma classe ClientUtility:

public class ClientUtility {
    private static Context initialContext;
    private static final String PKG_INTERFACES = "org.jboss.ejb.client.naming";

    public static Context getInitialContext() throws NamingException {
        if (initialContext == null) {
            Properties properties = new Properties();
            properties.put(Context.URL_PKG_PREFIXES, PKG_INTERFACES);
            initialContext = new InitialContext(properties);
         }
        return initialContext;
    }
}

Agora vamos criar a classe Client real que consumirá o bean que implantamos no servidor:

public class Client {
    
    //The lookup method to get the EJB name
    private static HelloWorldRemote doLookup() {
        Context context = null;
        HelloWorldRemote bean = null;
        try {
            // 1. Obtaining Context
            context = ClientUtility.getInitialContext();
            // 2. Generate JNDI Lookup name
            String lookupName = getLookupName();
            // 3. Lookup and cast
            bean = (HelloWorldRemote) context.lookup(lookupName);
 
        } catch (NamingException e) {
            e.printStackTrace();
        }
        return bean;
    }
 
    private static String getLookupName() {
        
         // The app name is the EAR name of the deployed EJB without .ear suffix.
         // Since we haven't deployed the application as a .ear, the app name for
         // us will be an empty string
         
        String appName = "";
 
        
         // The module name is the JAR name of the deployed EJB without the .jar
         // suffix.
        String moduleName = "ejb-remote-0.0.1-SNAPSHOT";
 
        
        // AS7 allows each deployment to have an (optional) distinct name. This
        // can be an empty string if distinct name is not specified.
        String distinctName = "";
 
        // The EJB bean implementation class name
        String beanName = "HelloWorldBean";
 
        // Fully qualified remote interface name
        final String interfaceName = "com.baeldung.ejb.tutorial.HelloWorldRemote";
 
        // Create a look up string name
        String name = "ejb:" + appName + "/" + moduleName + "/" + distinctName
                + "/" + beanName + "!" + interfaceName;
        
        return name;
    }
}

A classe Client consome o bean e gera o resultado.

7. Conclusão

Então criamos um servidor EJB e um cliente que consome o serviço. O projeto pode ser executado em qualquer Application Server.
