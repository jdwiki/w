gen_server é um recurso importante do Erlang e requer alguns pré-requisitos para entender todos os aspectos dessa funcionalidade:

  • Loop, recursão e estado
  • Processos de desova
  • Mensagem passando
  • princípios OTP

Uma boa maneira de aprender mais sobre um recurso em Erlang é ler diretamente o código-fonte do repositório oficial do github. O comportamento gen_server incorpora muitas informações úteis e estrutura interessante em seu núcleo.

gen_server é definido em gen_server.erl e sua documentação associada pode ser encontrada em stdlib Erlang documentation. gen_server é um recurso OTP e mais informações também podem ser encontradas em OTP Design Principles and User’s Guide.

Frank Hebert também oferece outra boa introdução ao gen_server de seu livro online gratuito Learn You Some Erlang for great good!

Serviço de saudação #

Usando o comportamento gen_server #

Um gen_server é uma máquina de estado finito específica que funciona como um servidor. gen_server pode lidar com diferentes tipos de eventos:

  • solicitação síncrona com handle_call
  • solicitação assíncrona com handle_cast
  • outra mensagem (não definida na especificação OTP) com handle_info

As mensagens síncronas e assíncronas são especificadas no OTP e são simples tuplas marcadas com qualquer tipo de dados.

Um simples gen_server é definido assim:

-module(simple_gen_server).
-behaviour(gen_server).
-export([start_link/0]).
-export([init/1, handle_call/3, handle_cast/2, handle_info/2,
         terminate/2, code_change/3]).

start_link() ->
    Return = gen_server:start_link({local, ?MODULE}, ?MODULE, [], []),
    io:format("start_link: ~p~n", [Return]),
    Return.

init([]) ->
    State = [],
    Return = {ok, State},
    io:format("init: ~p~n", [State]),
    Return.

handle_call(_Request, _From, State) ->
    Reply = ok,
    Return = {reply, Reply, State},
    io:format("handle_call: ~p~n", [Return]),
    Return.

handle_cast(_Msg, State) ->
    Return = {noreply, State},
    io:format("handle_cast: ~p~n", [Return]),
    Return.

handle_info(_Info, State) ->
    Return = {noreply, State},
    io:format("handle_info: ~p~n", [Return]),
    Return.

terminate(_Reason, _State) ->
    Return = ok,
    io:format("terminate: ~p~n", [Return]),
    ok.

code_change(_OldVsn, State, _Extra) ->
    Return = {ok, State},
    io:format("code_change: ~p~n", [Return]),
    Return.

Este código é simples: cada mensagem recebida é impressa na saída padrão.

comportamento do gen_server #

Para definir um gen_server, você precisa declará-lo explicitamente em seu código-fonte com -behaviour(gen_server). Observe que behavior pode ser escrito em US (comportamento) ou Reino Unido (comportamento).

start_link/0 #

Esta função é um atalho simples para chamar outra função: gen_server:start_link/3,4.

link_inicial/3,4 #

start_link() ->
    gen_server:start_link({local, ?MODULE}, ?MODULE, [], []).

Esta função é chamada quando você deseja iniciar seu servidor vinculado a um supervisor ou outro processo. start_link/3,4 pode registrar automaticamente seu processo (se você acha que seu processo precisa ser único) ou pode simplesmente gerá-lo como um processo simples. Quando chamada, esta função executa init/1.

Esta função pode retornar estes valores de definição:

  • {ok,Pid}
  • ignorar
  • {erro,Erro}

init/1 #

init([]) ->
    State = [],
    {ok, State}.

init/1 é a primeira função executada quando seu servidor será lançado. Este inicializa todos os pré-requisitos do seu aplicativo e retorna o estado para o processo recém-criado.

Esta função pode retornar apenas estes valores definidos:

  • {ok,Estado}
  • {ok,Estado,Tempo limite}
  • {ok,Estado,hibernação}
  • {parar, Razão}
  • ignorar

A variável State pode ser tudo, (por exemplo, lista, tupla, proplists, mapa, registro) e permanecer acessível a todas as funções dentro do processo gerado.

handle_call/3 #

handle_call(_Request, _From, State) ->
    Reply = ok,
    {reply, Reply, State}.

gen_server:call/2 executa este callback. O primeiro argumento é sua mensagem (_Request), o segundo é a origem da requisição (_From) e o último é o estado atual (State) do seu comportamento gen_server em execução.

Se você quiser uma resposta ao chamador, handle_call/3 precisa retornar uma destas estruturas de dados:

  • {responder,Responder,NewState}
  • {responder,Responder,NewState,Tempo limite}
  • {responder,Responder,NewState,hibernar}

Se você não quiser responder ao chamador, handle_call/3 precisa retornar uma destas estruturas de dados:

  • {noreply,NewState}
  • {noreply,NewState,Timeout}
  • {noreply,NewState,hibernate}

Se você quiser parar a execução atual do seu gen_server atual, handle_call/3 precisa retornar uma destas estruturas de dados:

  • {stop,Razão,Responder,NewState}
  • {parar,Motivo,NewState}

handle_cast/2 #

handle_cast(_Msg, State) ->
    {noreply, State}.

gen_server:cast/2 executa este callback. O primeiro argumento é sua mensagem (_Msg), e o segundo o estado atual do seu comportamento gen_server em execução.

Por padrão, essa função não pode enviar dados para o chamador, portanto, você tem apenas duas opções, continue a execução atual:

  • {noreply,NewState}
  • {noreply,NewState,Timeout}
  • {noreply,NewState,hibernate}

Ou pare seu processo gen_server atual:

  • {parar,Motivo,NewState}

handle_info/2 #

handle_info(_Info, State) ->
    {noreply, State}.

handle_info/2 é executado quando uma mensagem OTP não padrão vem do mundo exterior. Este não pode responder e, como handle_cast/2 pode fazer apenas 2 ações, continuando a execução atual:

  • {noreply,NewState}
  • {noreply,NewState,Timeout}
  • {noreply,NewState,hibernate}

Ou pare o processo gen_server em execução:

  • {parar,Motivo,NewState}

terminar/2 #

terminate(_Reason, _State) ->
    ok.

terminate/2 é chamado quando ocorre um erro ou quando você deseja encerrar seu processo gen_server.

code_change/3 #

code_change(_OldVsn, State, _Extra) ->
    {ok, State}.

A função code_change/3 é chamada quando você deseja atualizar seu código em execução.

Esta função pode retornar apenas estes valores definidos:

  • {ok, NewState}
  • {erro, Razão}

Iniciando este processo #

Você pode compilar seu código e iniciar o simple_gen_server:

simple_gen_server:start_link().

Se você quiser enviar uma mensagem para o seu servidor, você pode usar estas funções:

% will use handle_call as callback and print:
%   handle_call: mymessage
gen_server:call(simple_gen_server, mymessage).

% will use handle_cast as callback and print:
%   handle_cast: mymessage
gen_server:cast(simple_gen_server, mymessage).

% will use handle_info as callback and print:
%   handle_info: mymessage
erlang:send(whereis(simple_gen_server), mymessage).

Banco de dados simples de chave/valor #

Este código-fonte cria um serviço simples key/value store baseado na estrutura de dados map Erlang. Primeiramente, precisamos definir todas as informações referentes ao nosso gen_server:

-module(cache).
-behaviour(gen_server).

% our API
-export([start_link/0]).
-export([get/1, put/2, state/0, delete/1, stop/0]).

% our handlers
-export([init/1, handle_call/3, handle_cast/2, handle_info/2,
         terminate/2, code_change/3]).

% Defining our function to start `cache` process:

start_link() ->
    gen_server:start_link({local, ?MODULE}, ?MODULE, [], []).

%%%%%%%%%%%%%%
% API

% Key/Value database is a simple store, value indexed by an unique key. 
% This implementation is based on map, this datastructure is like hash 
# in Perl or dictionaries in Python. 

% put/2
% put a value indexed by a key. We assume the link is stable 
% and the data will be written, so, we use an asynchronous call with 
% gen_server:cast/2.

put(Key, Value) ->
    gen_server:cast(?MODULE, {put, {Key, Value}}).

% get/1
% take one argument, a key and will a return the value indexed 
% by this same key. We use a synchronous call with gen_server:call/2.

get(Key) ->
    gen_server:call(?MODULE, {get, Key}).

% delete/1 
% like `put/1`, we assume the data will be removed. So, we use an 
% asynchronous call with gen_server:cast/2.

delete(Key) ->
    gen_server:cast(?MODULE, {delete, Key}).

% state/0 
% This function will return the current state (here the map who contain all 
% indexed values), we need a synchronous call.

state() ->
    gen_server:call(?MODULE, {get_state}).

% stop/0
% This function stop cache server process.

stop() ->
    gen_server:stop(?MODULE).

%%%%%%%%%%%%%%%
% Handlers

% init/1
% Here init/1 will initialize state with simple empty map datastructure.

init([]) ->
    {ok, #{}}.

% handle_call/3
% Now, we need to define our handle. In a cache server we need to get our 
% value from a key, this feature need to be synchronous, so, using 
% handle_call seems a good choice:

handle_call({get, Key}, From, State) ->
    Response = maps:get(Key, State, undefined),
    {reply, Response, State};

% We need to check our current state, like get_fea

handle_call({get_state}, From, State) ->
    Response = {current_state, State},
    {reply, Response, State};

% All other messages will be dropped here.

handle_call(_Request, _From, State) ->
    Reply = ok,
    {reply, Reply, State}.

% handle_cast/2
% put/2 will execute this function.

handle_cast({put, {Key, Value}}, State) ->
    NewState = maps:put(Key, Value, State),
    {noreply, NewState};

% delete/1 will execute this function.

handle_cast({delete, Key}, State) ->
    NewState = maps:remove(Key, State),
    {noreply, NewState};

% All other messages are dropped here.

handle_cast(_Msg, State) ->
    {noreply, State}.

%%%%%%%%%%%%%%%%
% other handlers

% We don't need other features, other handlers do nothing.

handle_info(_Info, State) ->
    {noreply, State}.

terminate(_Reason, _State) ->
    ok.

code_change(_OldVsn, State, _Extra) ->
    {ok, State}.

Usando nosso servidor de cache #

Agora podemos compilar nosso código e começar a usá-lo com erl.

% compile cache
c(cache).

% starting cache server
cache:start_link().

% get current store
% will return:
%   #{}
cache:state().

% put some data
cache:put(1, one).
cache:put(hello, bonjour).
cache:put(list, []).

% get current store
% will return:
%   #{1 => one,hello => bonjour,list => []}
cache:state().

% delete a value
cache:delete(1).
cache:state().
%   #{1 => one,hello => bonjour,list => []}

% stopping cache server
cache:stop().
